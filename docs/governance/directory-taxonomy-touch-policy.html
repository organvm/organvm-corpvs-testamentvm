Thesis

Every filesystem path encodes three independent claims: how risky it is to edit, who legitimately owns it, and which convention defined it. Most homedir drift is a conflation of those axes — config shipped to ~/Documents/, cache where state belongs, user-dev surfaces treated as if they were vendor territory.

Most OSes hand you 50+ such paths and expect you to internalize the taxonomy by osmosis. This page makes it explicit, then applies it.

The schema

Axis 1 — Touch policy (color)

TierLabelMeaning
🟥 NNEVER TOUCHOS-protected. SIP / WRP / root-owned virtual filesystem. Edits either physically blocked or break the system.
🟧 AADMIN ONLYRequires sudo / Administrator / elevation. Legitimate edits possible but always intentional.
🟨 XAPP-MANAGEDAn app owns this; it writes, you generally don't. Hand-edit only when fixing something specific.
🟩 UUSER FREELYYour daily work surface. Touch without ceremony.
🟦 DDEV FREELYDev scope; touch without ceremony, but with awareness (track in dotfiles, mind the cascade).
🟪 HHYBRIDMultiple roles legitimately edit (e.g., /etc/hosts — admin owns; both admin and dev edit).
⬛ VVIRTUALNot a real file. Reading reports state; writing changes kernel/runtime state. (Linux /proc/, /sys/.)

Axis 2 — Role owner (tag prefix)

[OS]OS maintainerVendor (Apple, Microsoft, distro).
[ADM]Admin / installerYou-with-sudo, or installer software.
[USR]UserYou, without elevation.
[DEV]User-devYou, with dev tooling.

Axis 3 — Convention origin (tag suffix)

(HIG)Apple HIGmacOS Human Interface Guidelines.
(XDG)XDGXDG Base Directory Specification.
(FHS)FHSFilesystem Hierarchy Standard (Linux).
(NT)Windows NTMicrosoft convention.
(Legacy)LegacyPre-XDG Unix dotfiles.
(Custom)CustomUser-invented; no standard.

A combined tag reads as policy + role + convention:

  • 🟥 N [OS] (HIG) — vendor-owned, never touch, macOS-specific.
  • 🟦 D [DEV] (XDG) — dev-owned, touch freely, modern XDG.
  • 🟪 H [ADM] (FHS) — admin-owned but multi-role-edited, Linux FHS.
Why three axes
  • Color and tag are doing different work. Color signals what happens if you touch it wrong; tag signals who legitimately owns it. Conflating them loses information: /etc/ and ~/.config/ have similar semantic role (declarative config) but very different touch risk (sudo vs free).
  • Convention tags exist because the same role can have multiple historical homes. ~/.gitconfig (Legacy) and ~/.config/git/config (XDG) are both legitimate; knowing which lineage a path belongs to predicts whether other tools will look there.

Three roles

RoleWhat they doPrivilege
OS maintainerInstalls / updates the OS itself, system packages, drivers, system services.Root / Administrator / sudo
UserDay-to-day file creation, GUI settings, app installs through installers or stores.Own home only (no root)
User-devAll of "user" + installs dev tools, manages source, writes scripts, edits configs in text.Home + selective sudo for system-scope installs

On a personal machine, all three roles are the same person. The territory is what's distinct.

The ideal use of each canonical directory. "What is this directory for?"

XDG (modern Unix)

~/.config/🟦 DDeclarative config — hand-editable text describing how tools behave.
~/.local/share/🟦 DApp data — generated files apps own and keep. Backup-worthy.
~/.local/state/🟦 DState — logs, undo history, "last-X" memory. Not config; not cache.
~/.cache/🟦 DDisposable cache — rm -rf should never break anything important.
~/.local/bin/🟦 DUser-installed binaries on PATH.

~/.zshrc, .zshenv, .editorconfig🟦 DShell + universal config — paths the tool hardcodes.

Three philosophies, one topology
  • All three OSes share the same structural pattern: reserved kernel territory, shared installer territory, per-user sandbox. The labels differ; the topology doesn't.
  • "User-dev" is a cultural construct on macOS and Windows (the OS doesn't formally recognize it), but is almost the default personal-machine mode on Linux. This asymmetry shapes how leakage between user and dev territory happens.
  • ⬛ V is unique to Linux at any meaningful scale. /proc/ and /sys/ are entire universes of "files" that aren't files. macOS hides equivalent functionality in syscalls; Windows in the Registry. This is a real architectural difference worth tagging distinctly.

Highest-information findings
  • The single most diagnostic fact: ~/Code/ is 53 GB and ~/Workspace/ is 11 GB. That's not "I tried two conventions" — that's a real operational split. Before consolidating, surface why the split exists; it may be load-bearing.
  • The four empty-stub dev dirs (~/Projects/, ~/Development/, ~/Studio/, ~/WORK/, all 0 B) reveal a different drift: past organizational attempts that never took root. Empty namespace claims are easier to remove than competing repositories, but they pollute mental search-space at the same rate.
  • The expansive-inquiry-* sprawl is a symptom of a missing intermediate concept — they want to be grouped, but no parent directory exists yet to hold them. Choosing the parent name is itself an ontological act.
  • Apple's ~/Developer/ getting the hammer icon is a tiny detail but real — if you want a dev-dir choice with external convention backing, it's the only one that exists on macOS. Picking ~/Code/ chooses aesthetic consistency with ~/Documents/ over Apple's blessing; defensible, just a tradeoff worth knowing.

Regenerate this audit

The findings above are point-in-time. To re-run them on current disk state:

# Dev-dir sizes (severe finding)
du -sh ~/Code ~/Workspace ~/Projects ~/Development ~/Studio ~/WORK 2>/dev/null

# Loose files at home root
find ~ -maxdepth 1 -type f ! -name ".*" | sort

# Hidden backup files at home root
ls -la ~/.claude.json.bak.* 2>/dev/null

# expansive-inquiry-* sprawl count
ls -d ~/expansive-inquiry-*/ 2>/dev/null | wc -l

# Apple-blessed dev dir presence
test -d ~/Developer && echo "✓ Developer present (hammer icon)" || echo "✗ Developer missing"

# Cruft-candidate dotdirs (size-sorted)
du -sh ~/.* 2>/dev/null | sort -hr | head -20

Visual hardening across surfaces

Color tags in a markdown table are step one — propagating them across surfaces is step two. The schema hardens by repetition.

Terminal — EZA_COLORS / LS_COLORS

export EZA_COLORS="di=34:ow=34:tw=34"   # dir = blue

Prompt — Starship substitutions

# ~/.config/starship.toml
[directory]
substitutions = {
  "~/.config" = "[🟦 config]",
  "~/.local"  = "[🟦 local]",
  "/etc"      = "[🟪 etc]"
}

IDE — VS Code / Cursor workspace colors

// .vscode/settings.json
{
  "workbench.colorCustomizations": {
    "titleBar.activeBackground": "#2370a3",  // 🟦 dev project
    "titleBar.activeForeground": "#ffffff"
  }
}

Finder — native colored tags (macOS)

tag -a Red ~/2/                                 # 🟥 DO NOT TOUCH (or delete)
tag -a Orange ~/workspace-autopsy-governance\ 2/  # 🟧 review & merge

Documentation — propagate the legend

Add the color squares to every README, CLAUDE.md, plan, or ontology doc that lists paths. The schema only hardens if it appears everywhere the concept appears — 🟥 N becomes recognizable as "kernel territory" only after the third or fourth time you see it.

Two minimal rules that travel across all three OSes

  1. If your package manager put a file there, don't hand-edit it. Apt, dnf, brew, choco, scoop, the Mac App Store — they all expect to own certain dirs and will overwrite or break on conflict. Use them, don't fight them.
  2. If you didn't create the dir and don't know what's in it, leave it alone. Especially for hidden / system / virtual dirs. The default answer to "what does /sys/class/block/ do" is "nothing of yours."
