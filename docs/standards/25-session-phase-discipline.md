## 1. Purpose & Scope

This document codifies the **session-phase discipline** — a ten-phase ceremony that walks any substantive work item from initial scope through merged, archived completion. The discipline exists because:

- **Direct-edit-then-push loses the audit trail**: PR comments, CI checks, review history, squash-commit messages all encode *why* changes happened. Without ceremony, that context evaporates.
- **The lifecycle has natural decision points**: explore→plan crosses the FRAME→SHAPE boundary; verify→push crosses PROVE→SHIP. Skipping a phase doesn't speed things up; it just hides the decision behind opaque commits.
- **Dogfood breeds discoverability**: when every substantive change ceremoniously walks the discipline, future sessions reading the git history can reconstruct intent, not just outcome.

The discipline is **convention, not enforcement**. None of the phases are mechanically required (no hard hook blocks). The cost of skipping is invisible — accumulated bookkeeping debt, lost audit trail, missing closure records. The cost of applying is ~3-5 minutes per cycle.

## 2. The 10-Phase Lifecycle

```
1. /s-01-explore   ← FRAME    survey the territory (read-only)
2. /s-02-plan      ← SHAPE    persist dated plan with premortem
3. /s-03-branch    ← BUILD    feature branch per § 5 naming
4. /s-04-code      ← BUILD    atomic commits, conventional-commits prefix
5. /verify         ← PROVE    built-in /verify or /run; observe behavior
6. /s-05-push      ← SHIP     push branch + open PR
7. /s-06-wait      ← SHIP     read-only status snapshot (CI/reviews)
8. /review         ← SHIP     built-in /review, /code-review, /security-review
9. /s-07-amend     ← SHIP     respond to feedback as NEW commits (never amend pushed history)
10. /s-08-merge    ← FINISH   squash + delete-branch
11. /closeout      ← FINISH   built-in skill: 10-index, working-state capture
12. /s-09-archive  ← FINISH   gate-check + move session to archived storage
```

`/s-00-phase` is the **meta-detector** — invoke any time to determine current state and get the next-phase suggestion.

Each phase has a corresponding user-level slash command at `~/.claude/commands/s-NN-{phase}.md`. Built-in phases (verify, review, closeout) are skills shipped with Claude Code. The canonical schema at `~/.config/ai-context/session-phases.yaml` declares the lifecycle once; per-agent adapters (`~/.codex/policy/session-phases.md`, `~/.gemini/policies/session-phases.json`) are auto-regenerated by `~/.config/ai-context/scripts/regen-session-policies`.

## 3. When to Apply

Apply the discipline for **substantive work**:

- New features, bug fixes, refactors that touch >50 lines or cross file boundaries
- Architectural changes (new modules, new conventions, schema modifications)
- Any work that benefits from a "why was this done?" record (most of it)
- Any work that touches files multiple sessions will need to read

**Skip for trivial edits**:

- Typo fixes inside a single file
- One-line config changes that are obviously correct
- Sibling-session auto-sync writes (hooks already handle these)
- Memory file updates inside `~/.claude/projects/.../memory/` (the chezmoi auto-sync + additive policy govern these)

The judgment call is "would a reasonable reviewer want to see this in a PR?" — if yes, ceremony. If no, direct-edit is fine.

## 4. The Dogfood Principle

**The conductor (or any agent) doing substantive work applies the discipline to the work they're shipping, including the discipline itself.**

This document is the canonical example: it was authored via the ceremony it codifies. The plan file at `<corpvs>/.claude/plans/2026-05-28-session-phase-discipline-sop.md` was committed first; the SOP went on a `feature/session-phase-discipline-sop` branch off `origin/main`; the PR went through review and squash-merge; the closeout summary captured the cycle.

The dogfood is not theatrical — it's the most efficient validation. If the ceremony doesn't work for documenting itself, it won't work for anything else.

## 6. Hard Rules (verbatim cross-reference)

These rules from `17-branch-governance.md § 10` apply throughout the SHIP and FINISH phases:

1. NEVER force-push to main/master/develop on any repo, any tier.
2. NEVER perform a direct `git push origin main` on a *protected* branch (the Rule #12 scope-bound clarification permits direct-push on unprotected branches — verify via `gh api .../branches/<branch>/protection`).
3. NEVER auto-amend code in response to review comments. Code changes ALWAYS require human-in-the-loop.
4. NEVER post substantive comment replies in the author's name. Metadata-only replies allowed per `~/.claude/scheduled-tasks/canonical-replies.md`.
5. NEVER merge a PR with unanswered maintainer-other-than-author comments.
6. NEVER act on a PR whose title contains `WIP`, `DRAFT`, `[wip]`, `[draft]`, or `DO NOT MERGE`.
7. NEVER act on a Tier-1 repo beyond reporting.
