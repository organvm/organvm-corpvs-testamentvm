## Auto-fix posture (v2.1.0 — flipped from "never")

This routine **auto-fixes** via worktree→PR→merge (standing policy `feedback_always_auto_fix_via_worktree_pr`; supersedes the prior read-only "conductor decides" rule). A report is a fraction of the battle — diagnose the *direction* of divergence, then drive the fix to the finish line. Best practices are non-negotiable: **never commit a secret** (scan the diff; app configs carry live tokens — see the codex `Authorization`/`bearer_token_env_var` case), respect blocking gates rather than routing around them, and route every source mutation through a PR on `~/Workspace/4444J99/domus-semper-palingenesis`.

Three divergence directions, three fixes:

1. **Source ahead of deployed** (a merged change was never applied — what the 2026-06-05 run hit). Not "drift" in the classic sense; the shim reports it indirectly (stale behavior). Fix = `chezmoi apply <deployed-path>` (target-only; no source mutation, no PR needed). Re-run to confirm.
2. **Deployed ahead of source, change is GOOD** (an intentional edit made directly to the deployed file). Fix = lift into source: create a worktree, `chezmoi add -S <worktree> <deployed-path>` (or `chezmoi edit`), commit, PR, merge, then `chezmoi apply`.
3. **Deployed drifted from source accidentally.** Fix = revert deployed: `chezmoi apply --force <deployed-path>`. No PR (source already canonical).

For excluded-by-design rows reporting drift: that should not happen (they're unmanaged). If one does, the `.chezmoiignore` entry was likely removed — investigate before acting.

Pair each report with the most recent commit to the relevant chezmoi-source file — if the source was edited within the drift window, the change may be in-flight; reconcile rather than fight it.
